# 项目问题清单与改进建议

## 一、P0 级问题（影响正确性，必须修复）

### 1. `position_floor_new` 配置不一致

- **文件**：`pipeline/task4_strategy.py` 第 187 行
- **问题**：`getattr(config, "position_floor_new", 0.15)` 返回默认值 0.15，但 `config.yaml` 中 `single_position_min` 配置为 0.2，两者不一致。实际运行中仓位下限使用了 0.15 而非预期的 0.2。
- **影响**：资金分配比例与配置预期不符，可能导致提交结果异常。

### 2. `allocate_positions` 仓位归一化逻辑缺陷

- **文件**：`pipeline/task4_strategy.py` 第 183-196 行
- **问题**：仓位分配逻辑为 `clip → 归一化 → clip → 用 diff 修正第一个元素`。第二次 clip 后没有再次归一化，仅通过修正第一个元素来弥补差额，当多只股票同时被 clip 时，最终 `capital_ratio.sum()` 可能不等于 1.0。
- **影响**：资金比例之和可能不为 1，不符合比赛 `result.xlsx` 的格式要求。



### 3. `scoring.association` 配置未被使用

- **文件**：`config/config.yaml` 第 30-35 行 vs `pipeline/task2_relation_mining.py`
- **问题**：配置文件中定义了 `scoring.association.*` 关联权重，但 `task2_relation_mining.py` 中使用的是硬编码的 `WEIGHT_PROFILES`，从未读取配置值。
- **影响**：配置与实际行为不一致，调参时修改配置文件无效。

## 二、P1 级问题（影响健壮性和比赛评分）

### 5. 交易日历兜底逻辑无法处理中国节假日

- **文件**：`pipeline/fetch_data.py` 第 181 行
- **问题**：`pd.bdate_range` 仅排除周末，不排除春节、国庆等中国股市休市日。
- **影响**：在降级路径下可能生成错误的交易日列表，导致"在非交易日买入"的逻辑错误。

### 6. 缺少预测模型性能评估实验

- **对应题目要求**：Task3 —— "自行设计实验分析所构建的预测模型的性能"
- **现状**：`backtest.py` 仅计算投资总收益和净值曲线，没有对 CAR 预测模型的准确率进行评估。
- **影响**：不完全符合比赛题目 Task3 的要求，分析报告中缺少模型性能分析章节。

### 7. 缺少典型事件完整展示

- **对应题目要求**：Task2 —— "选择一个典型事件，展示实现过程"
- **现状**：`report_builder.py` 生成的是通用周报，没有针对某个典型事件从识别→关联→预测→选股的完整链路展示。
- **影响**：不完全符合比赛题目 Task2 的要求。

### 8. `event_taxonomy` 三处重复

- **位置**：
  - `config/config.yaml` 中的 `event_taxonomy` 节
  - `pipeline/task1_event_identify.py` 中的 `DEFAULT_EVENT_TAXONOMY`
  - `pipeline/models.py` 中 `AppConfig.event_taxonomy` 的默认值
- **问题**：三处内容基本一致但独立维护，修改一处容易忘记同步其他两处。
- **影响**：维护成本高，容易出现配置漂移。

### 9. 过于宽泛的异常捕获

- **文件**：`pipeline/fetch_data.py`（多处）
- **问题**：使用裸 `except Exception:` 吞掉异常，不记录堆栈信息。
- **影响**：出现问题时难以定位根因，调试困难。

### 10. `load_imported_event_records` 硬错误导致流程中断

- **文件**：`pipeline/fetch_data.py` 第 314 行
- **问题**：`raise RuntimeError(...)` 在单个事件文件格式错误时直接中断整个流程。
- **影响**：容错性差，一条坏数据导致全部失败。

## 三、P2 级问题（影响性能、可维护性和代码质量）

### 11. Task1 事件聚类 O(n²) 性能问题

- **文件**：`pipeline/task1_event_identify.py` 第 46-63 行
- **问题**：双层 `iterrows()` 嵌套循环计算相似度，时间复杂度 O(n²)。
- **影响**：当新闻量达到数千条时，运行时间显著增加，可能导致超时。

### 12. Task2 关联计算笛卡尔积

- **文件**：`pipeline/task2_relation_mining.py` 第 58-86 行
- **问题**：`event_df.iterrows()` × `stock_df.iterrows()` 全量遍历。
- **影响**：事件和股票数量较多时计算量爆炸。

### 13. 多处 `apply(axis=1)` 未向量化

- **文件**：`pipeline/task4_strategy.py` 第 49-52 行、第 55-59 行
- **问题**：使用 `df.apply(lambda row: ..., axis=1)` 计算 `pass_basic_filter`、`is_tradeable`、`momentum_5d`。
- **影响**：比向量化操作慢 10-100 倍。

### 14. 大量硬编码阈值和魔法数字

| 位置 | 硬编码值 | 说明 |
|------|---------|------|
| `task1_event_identify.py:59` | `similarity >= 0.18`, `title_similarity >= 0.35` | 聚类相似度阈值 |
| `task1_event_identify.py:73` | `0.3*heat + 0.35*intensity + 0.2*scope + 0.15*sentiment` | 综合评分权重 |
| `task1_event_identify.py:74` | `logistic(6 * (raw - 0.5))` | 缩放因子 |
| `task1_event_identify.py:144` | 硬编码 7 只军工股票名称 | 范围评分逻辑 |
| `task2_relation_mining.py:95` | `0.30`, `0.15` | 业务匹配分数 |
| `task2_relation_mining.py:115-136` | `INDUSTRY_GROUP_MAP` 完整字典 | 行业映射硬编码 |
| `task3_impact_estimate.py:64` | `* 2.5`, `min(0.25, max(0.10, ...))` | 自适应缩放边界 |
| `task3_impact_estimate.py:82` | `/ 600` | 流动性归一化分母 |
| `task3_impact_estimate.py:87` | `0.55 + market_state` | 市场状态偏置 |
| `task3_impact_estimate.py:95` | `0.22 + 0.18 * liquidity` | CAR 时间分配系数 |
| `task4_strategy.py:82` | `min_score_threshold = -0.01` | 基础过滤阈值（注释写"可配置"但未实现） |
| `backtest.py:68-70` | `commission_rate = 0.001`, `slippage = 0.0005` | 交易成本 |

### 15. 使用 print 代替 logging

- **文件**：全项目
- **问题**：使用 `print(f"[WARN] ...")` / `print(f"[ERROR] ...")` 输出日志，无日志级别控制、时间戳。
- **影响**：日志不可控，生产环境无法通过级别过滤关闭调试信息。

### 16. 关键数据流转节点无日志

- **文件**：`pipeline/workflow.py`
- **问题**：各步骤之间传递的 DataFrame 没有记录数据规模变化（行数、关键列统计）。
- **影响**：出问题时无从判断是哪一步数据丢失或异常。

### 17. 回测引擎重复获取价格数据

- **文件**：`pipeline/backtest.py` 第 44-50 行
- **问题**：每周迭代时都调用 `fetch_price_history` 重新获取价格数据，而 `run_weekly_pipeline` 内部也已获取过。
- **影响**：重复 API 调用，浪费时间并可能触发 API 限流。

### 18. `task3_impact_estimate.py` 中 merge 可能导致静默数据丢失

- **文件**：`pipeline/task3_impact_estimate.py` 第 29-37 行
- **问题**：`relation_df.merge(event_df, ...)` 和 `merge(stock_df, ...)` 如果股票代码格式不一致（如 `600000.SH` vs `600000`），会静默丢失数据。
- **影响**：部分关联股票在影响预测阶段被遗漏。

### 19. `next_trading_date` 日期查找逻辑隐患

- **文件**：`pipeline/task4_strategy.py` 第 121-132 行
- **问题**：`while target.weekday() != target_weekday` 从 `asof_date` 开始查找，但如果 `asof_date` 本身就是目标星期几，不会往后找。`for offset in range(5)` 最多往后找 5 天，可能超出范围提前 break。
- **影响**：极端情况下可能找不到正确的交易日。

### 20. `subject_bias` 映射硬编码

- **文件**：`pipeline/task3_impact_estimate.py` 第 209-215 行
- **问题**：事件主体偏置映射写死在代码中，未在配置文件中定义。
- **影响**：与项目"配置驱动"的设计理念不一致，调整偏置需要改代码。
