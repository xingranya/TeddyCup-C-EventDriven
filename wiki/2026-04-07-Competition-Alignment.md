# 2026-04-07 — 竞赛对齐修复

本次更新的重点不是扩大功能面，而是把项目里与泰迪杯 C 题要求不够对齐、或者会让提交结果失真的地方收紧，确保“结果可信、报告对题、配置生效”。

## 本次修复了哪些与竞赛题目对不上的地方

### 1. Task 1：事件分类体系真正配置驱动

- 之前虽然 `config/config.yaml` 里有 `event_taxonomy`，但周度流程没有把它真正传给 `task1_event_identify.py`
- 现在 `workflow.py` 会显式把 `config.event_taxonomy` 注入 Task 1
- 含义：事件分类体系终于和项目配置保持一致，不会出现“改了配置但分类逻辑没变”的假配置现象

### 2. Task 2：关联强度权重不再是改不动的硬编码

- 之前 `config.yaml` 里定义了 `scoring.association`，但 `task2_relation_mining.py` 实际仍在使用硬编码权重
- 现在基础权重改为读取 `scoring.association`
- 若按事件主体类型做差异化处理，则通过 `scoring.association_profiles` 进行配置化倍率调整
- 含义：Task 2 的“关联强度指标体系”终于能和文档、配置、实际代码一致

### 3. Task 2 / Task 3：报告补齐了典型事件展示与模型性能实验

- 赛题要求：
  - Task 2 需要“选择一个典型事件，展示实现过程”
  - Task 3 需要“自行设计实验分析所构建预测模型的性能”
- 之前周报只有通用结果表，没有单独把这两部分落出来
- 现在 `report_builder.py` 新增：
  - “典型事件完整展示”
  - “模型性能实验”
  - “数据来源与限制”
- 含义：现在自动生成的 `report.md` 更接近竞赛平台需要的分析报告结构

### 4. Task 3：性能实验去掉了未来数据泄漏

- 之前模型性能实验会扫描 `outputs/weekly` 下所有周目录
- 如果重跑旧周，而仓库里已经有更新周的输出，就会把未来实现结果算进当前报告
- 现在性能实验只会统计 `<= 当前 asof_date` 的历史周目录
- 含义：实验结果不再带前视偏差，报告里的准确率、Top-K 命中率和秩相关更可信

### 5. Task 4：仓位分配严格满足比赛提交约束

- 之前仓位逻辑存在两类问题：
  - 代码里读过一个幽灵配置 `position_floor_new`
  - 补差式归一化与四舍五入可能造成配置漂移
- 现在仓位分配改为：
  - 先按得分归一化
  - 再做上下限约束分配
  - 最后用最大余数法舍入
- 严格使用 `single_position_min` / `single_position_max`
- 含义：更稳定地满足“最多 3 只股票 + 资金比例求和为 1”的提交要求

### 6. 报告不再误报未生成产物

- 之前增强模块失败时，报告仍可能列出不存在的 `event_study` / `kg_visual` 文件名
- 现在报告只列真实存在的增强文件；如果没生成，会明确写“本次未生成……产物”
- 含义：交付报告对用户和评审都更诚实，不会出现“报告写有图，但目录里没有”的错位

### 7. 比赛周数据导入更稳

- 之前单条导入事件坏数据可能导致整批流程中断
- 现在改为：
  - 异常记录写 warning
  - 坏记录跳过
  - 只有完全没有有效事件时才整体失败
- 含义：更适合比赛周临近提交时快速迭代导入事件，不会因为一条坏数据把整周运行打断

## 对竞赛交付的直接影响

- `result.xlsx` 的仓位比例更稳定，更贴合提交格式要求
- `report.md` 现在能自动覆盖典型事件展示、模型性能实验和数据来源说明
- 分类体系、关联权重、主体偏置、空仓阈值等关键策略参数不再只藏在代码里
- 旧周复盘时不会把未来周结果混进性能实验

## 本次没有刻意扩大的范围

- 没有引入新的重型爬虫体系
- 没有更换整套预测框架
- 没有处理 Task 1 O(n²) 聚类、Task 2 全量笛卡尔积这类第二梯队性能问题

本次的目标很明确：先把现有方案做成一个更能交卷、更能解释、也更不容易失真的竞赛版本。
